# Internationalisierung (i18n)

## Grundprinzip

UI-Strings dürfen **niemals** direkt in Komponenten stehen — weder Deutsch noch Englisch.
Alle Texte werden über die i18n-Funktion `t('key')` eingebunden.

**Verboten:**
```tsx
<button>Speichern</button>
<button>Save</button>
```

**Korrekt:**
```tsx
<button>{t('actions.save')}</button>
```

---

## Ebene 1 — UI-Strings externalisieren

### Regel: t() für alle user-facing Strings

Betrifft: Buttons, Labels, Überschriften, Fehlermeldungen, Hilfetexte,
Platzhaltertexte, Tooltips, Bestätigungsdialoge, Status-Badges.

Ausgenommen: technische IDs, API-Keys, Dateinamen.

### Key-Konvention

```
{namespace}.{element}
actions.save              → "Speichern" / "Save"
actions.cancel            → "Abbrechen" / "Cancel"
actions.confirm           → "Bestätigen" / "Confirm"
actions.reject            → "Verwerfen" / "Reject"
errors.generic            → "Ein Fehler ist aufgetreten." / "An error occurred."
errors.not_found          → "Nicht gefunden." / "Not found."
screens.{name}.title      → Seitentitel
screens.{name}.{element}  → Seitenspezifische Elemente
labels.{name}             → Formularfelder und Datenbeschriftungen
status.{value}            → Status-Beschriftungen (z.B. status.draft, status.linked)
```

### Platzhalter für fehlende Keys

Wenn ein Key noch nicht in den Übersetzungsdateien existiert:
```tsx
{t('TODO.narrative_editor.analyse_button')}
```
→ Muss als GitHub Issue erfasst werden (`i18n` Label), vor Release auflösen.

## Ebene 2 — Layout-Verhalten bei Sprachwechsel

### Deutscher Text ist länger

Deutsche Texte sind typischerweise **20–40% länger** als die englischen Entsprechungen.

| Englisch | Deutsch |
|---|---|
| Save | Speichern |
| Cancel | Abbrechen |
| Confirm | Bestätigen |
| Analysis workflow | Analyse-Workflow |
| Causal model editor | Wirkgefüge-Editor |

### Konsequenz: Kein festes width auf Textelementen

❌
```tsx
<button style={{ width: "100px" }}>…</button>
<span style={{ width: "80px" }}>…</span>
```

✅
```tsx
<button style={{ padding: "8px 16px" }}>…</button>
<span style={{ whiteSpace: "nowrap" }}>…</span>
```

### Konsequenz: Flex statt Grid für Text-Layouts

Wenn Elemente nebeneinander stehen und eines davon Text enthält:

❌
```tsx
<div style={{ display: "grid", gridTemplateColumns: "120px 1fr" }}>
  <span>{t('label.created_at')}</span>
  <span>{value}</span>
</div>
```

✅
```tsx
<div style={{ display: "flex", gap: "12px", alignItems: "baseline" }}>
  <span style={{ flexShrink: 0 }}>{t('label.created_at')}</span>
  <span>{value}</span>
</div>
```

### Konsequenz: min-width statt width für Badges und Chips

```tsx
// Erlaubt wachsen, verhindert Kollaps
<span style={{ minWidth: "60px", padding: "2px 8px" }}>
  {t('status.draft')}
</span>
```

## Übersetzungsdateien

Ablage: `frontend/src/i18n/`

```
frontend/src/i18n/
├── de.json     ← Primärsprache
└── en.json     ← Sekundärsprache
```

Neue Keys immer in **beide** Dateien eintragen.
Fehlende Übersetzung in einer Sprache → `TODO` als Wert, Issue anlegen.

---

## Regeln für den Frontend-Agenten

1. Jeder neue String bekommt einen Key — kein Text direkt im JSX
2. Keys sofort in beide Sprachdateien eintragen
3. Wenn ein Key unklar ist: zuerst den Key benennen, dann umsetzen — nicht umgekehrt
4. Kein `width` auf Elementen die lokalisierte Texte enthalten
5. Beim Test: mentale Probe mit 140% Textlänge — würde es immer noch passen?
